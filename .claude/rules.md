# Coding Rules & Best Practices

When interacting with the `finance-tracker` repository, adhere to the following rules based on the modern `backend/` strategy:

## Languages & Conventions
- **Python Version:** Use Python 3.11+ (Python 3.13 recommended).
- **Typing:** Strict type hinting is required (`dict[str, Any]`, `list[int]`, etc.), especially for FastAPI parameters and response models.
- **Framework Choices:**
  - HTTP definitions must happen in `FastAPI` Routers (`backend/app/routers/`).
  - Request/Response validation MUST happen via `Pydantic` models (`backend/app/schemas/`).
  - Database access SHOULD be abstracted via `SQLAlchemy` queries in `backend/app/services/`.

## Parsing Engine Upgrades
- **New Parsers:** If a new transaction origin or bank format is discovered, a parser should ideally be registered under `backend/ingestion/parsers/` with exact unit tests checking the pattern.
- **Mandatory Fields:** A valid transaction always needs an `amount`, `direction`, `provider`, and `account_id`.

## Operations
- **Migrations:** Modifying any SQLAlchemy models requires an autogenerated Alembic migration: `alembic revision --autogenerate -m "describe change"`, applied via `alembic upgrade head`.

## Boundaries
- Modifying the `/finance-tracker-legacy/` directory is generally out of scope unless migrating its legacy parsing rules to the new backend.
